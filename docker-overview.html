What is Docker?
Docker is a platform that lets you package and run software in isolated environments called containers. Think of it like a shipping container for code — it holds everything an app needs, and works the same anywhere.
The Core Problem It Solves
Ever heard "it works on my machine"? Docker eliminates that. By bundling your app with its exact dependencies, it runs identically on every developer's laptop, every test server, and every production environment.
Why Teams Adopt It
Faster onboarding, consistent environments, easy scaling, and reproducible deployments. A single docker-compose up command can spin up an entire multi-service application stack in seconds.

Core Concepts
01
Runtime Unit

CONTAINERS Isolated, lightweight process environments

A container is a running instance of an image. It's an isolated process on your machine that has its own filesystem, networking, and process space — but shares the host OS kernel. Far lighter than a virtual machine.

Startup Time
Milliseconds
vs VM
~10× lighter
Isolation
Process-level
Common Commands bash
docker run -d -p 8080:80 nginx # start docker ps # list running docker stop <id> # stop docker exec -it <id> bash # shell
Key properties
Immutable by defaultEphemeralPort-mappedNamespaced
→Containers are disposable. Never store important data inside a container — use volumes instead. Stopping a container does not delete it; docker rm does.
02
Blueprint

IMAGES Read-only snapshot of an environment

An image is the template your container is created from. It's a layered, read-only stack: a base OS, then runtime, then libraries, then your app code. Each layer is cached, so rebuilds are fast — only changed layers re-download.

Format
OCI Standard
Layers
Cached
Built from
Dockerfile
Image Commands bash
docker build -t myapp:1.0 . # build docker images # list local docker pull python:3.11-slim # download docker push myapp:1.0 # publish
Image naming
registry/name:taglatest = default tagDigest = immutable ID
→Prefer slim or alpine base images for AI workloads when possible. For GPU-dependent work, use official nvidia/cuda base images to ensure CUDA compatibility.
03
Build Script

DOCKERFILE Recipe for building your image

A Dockerfile is a plain-text recipe that tells Docker how to build your image step by step. Each instruction creates a new layer. Well-written Dockerfiles use multi-stage builds to keep final images small and secure.

File name
Dockerfile
Language
DSL
Each line
= 1 layer
Python AI App Example Dockerfile
FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt COPY . . EXPOSE 8000 CMD ["python", "main.py"]
Key instructions
FROMRUNCOPYENVEXPOSECMDENTRYPOINT
→Order matters for cache efficiency. Put frequently changing lines last (e.g. COPY app code after installing dependencies) so Docker reuses earlier cached layers on rebuild.
04
Persistent Storage

VOLUMES Data that survives container restarts

Containers are ephemeral — when removed, their internal data disappears. Volumes solve this by mounting external storage into the container. Critical for databases, model weights, training data, and logs.

Survives
rm / restart
Shareable
Multi-container
Managed by
Docker Engine
Volume Usage bash
# Named volume (Docker manages path) docker run -v model-data:/app/models myapp # Bind mount (your local folder) docker run -v $(pwd)/data:/app/data myapp
Use volumes for
Database filesModel weightsTraining datasetsLog outputConfig files
→For AI workloads, storing large model weights (e.g. LLM checkpoints) in a named volume means they download once and are reused across container restarts — saving significant time.
05
Service Communication

NETWORKS How containers talk to each other

Docker networking lets containers communicate securely without exposing ports to the outside world. Containers on the same network can reach each other by service name — no hard-coded IPs needed.

Default
Bridge
Discovery
By name
Isolation
Per-network
Network Example bash
docker network create ai-net # Both containers join same network docker run --network ai-net --name api myapi docker run --network ai-net --name db postgres # api can reach db at: http://db:5432
Network types
bridge (default)host (no isolation)overlay (Swarm/K8s)none (air-gapped)
→Always create custom networks rather than using the default bridge. Custom networks support DNS-based service discovery — containers find each other by name automatically.
06
Multi-Service Orchestration

COMPOSE Define entire stacks in one file

Docker Compose lets you define and run multi-container applications from a single YAML file. One command starts everything — your API, database, cache, message queue, and AI model server — fully wired together.

Command
compose up
File
compose.yml
Best for
Dev / staging
compose.yml — AI Stack yaml
services: api: build: . ports: ["8000:8000"] depends_on: [db, redis] model-server: image: ollama/ollama volumes: ["models:/root/.ollama"] db: image: postgres:16
Compose handles
Service dependenciesShared networksVolume definitionsEnv varsHealth checks
→Compose is the #1 tool for local AI development. Spin up a full stack — inference server, vector database, API, and UI — with a single docker compose up -d.
07
Image Distribution

REGISTRY Store, share and version your images

A registry is a storage and distribution service for Docker images. Docker Hub is the public default. Teams typically use private registries (AWS ECR, GCP Artifact Registry, GitHub Packages) to control access and versioning.

Public
Docker Hub
Cloud
ECR / GCR / ACR
Self-hosted
Harbor / Nexus
Publishing an Image bash
# Tag image for your registry docker tag myapp:1.0 registry.example.com/myapp:1.0 # Login and push docker login registry.example.com docker push registry.example.com/myapp:1.0
Best practices
Semantic versioningNever use :latest in prodScan for CVEsSign images
→For AI models shipped as containers, tag images with both version + model size (e.g. inference:2.1-7b) so teams can pull the right variant without guessing.
