Foundations

Weeks 1–3 · ~3 hrs/week
Starting Point

Get your environment set up and build enough conceptual understanding of how MacOS apps work so you can have intelligent conversations with AI. You don't need to memorise syntax — you need to understand structure.

✓
Week 01 MacOS Ecosystem & Developer Tools Setup 3 hrs ▼

Get your machine ready and understand the landscape. MacOS apps can be built with multiple frameworks — you'll learn why Swift/SwiftUI is the right choice for native apps, and what Xcode is. No code yet; just orientation.

Key Topics
Xcode installation Swift overview SwiftUI vs AppKit Simulator setup File system basics

✓
Week 02 Swift & SwiftUI Fundamentals (Conceptual) 3 hrs ▼

Learn enough Swift and SwiftUI to read code and give meaningful instructions to AI. Focus on understanding concepts — variables, functions, views, state — rather than writing from scratch. Think of it like reading architectural blueprints.

Key Topics
Variables & types Functions Views & Modifiers @State & @Binding VStack / HStack / List
Resources
🎓

⚡ AI Prompt Strategy

When confused by code: "I'm looking at this SwiftUI code [paste it]. Can you walk me through what each line does as if I'm a business analyst, not a developer?"

✓
Week 03 Version Control with Git & GitHub 2 hrs ▼

Every serious project needs version control. Git lets you save snapshots of your work and undo mistakes. GitHub Desktop makes this accessible without command-line knowledge. This is your safety net throughout the project.

Key Topics
Repositories Commit & push Branches GitHub Desktop README basics
Resources

🔧

⚡ AI Prompt Strategy

Ask AI to generate your commit messages: "I made these changes to my project [describe]. Write me a clear Git commit message following best practices."

Requirements & Planning

Weeks 4–6 · ~4 hrs/week
Your Strength Zone

This is where your BA background becomes a superpower. Define exactly what you're building before writing a line of code. A great PRD will become your primary input to AI — the better the document, the better the generated code.

✓
Week 04 Defining Your Application Idea & Scope 3 hrs ▼

Decide what you're building and ruthlessly scope it down to an MVP. Resist feature creep. A focused, finished app beats an ambitious unfinished one. Use AI to pressure-test your idea and identify scope risks early.

Key Topics
Problem statement Target user MVP definition Feature prioritisation (MoSCoW) Technical feasibility
Resources
📋
MoSCoW Prioritisation Framework
Free · Use existing BA knowledge — apply to app features
📄
Y Combinator's "Startup = MVP" principles (adapted for personal projects)
Free · paulgraham.com essays · Mindset reading
🎯 Deliverable: 1-page App Concept Brief — problem, user, core 3 features only
⚡ AI Prompt Strategy

"I want to build a MacOS app that [description]. Act as a senior product manager and challenge my scope. What am I overcomplicating? What's the true MVP? What technical constraints should I know about?"

✓
Week 05 Writing Your Product Requirements Document (PRD) 5 hrs ▼

Create the PRD that will drive your entire build. A great PRD for AI-assisted development includes functional requirements written as user stories, acceptance criteria, data models, and UI behaviour descriptions. The AI will reference this document constantly.

Key Topics
User stories Acceptance criteria Data model overview UI flow description Non-functional requirements Out of scope
Resources
📋
PRD Template for AI-Assisted Development (see AI tip)
Use Claude to generate a tailored PRD template for your specific app type
🎓
The Complete Product Management Course (Cole Mercer)
Udemy · Section on PRDs & User Stories only · ~2 hrs
🎯 Deliverable: Full PRD document (3–8 pages) covering all core features with user stories and acceptance criteria
⚡ AI Prompt Strategy

"I am building a MacOS app [paste concept brief]. Generate a PRD template structured specifically so that a developer — or an AI acting as a developer — could use it to write the code. Include sections for user stories, data model, UI behaviour, and edge cases."

✓
Week 06 UI Wireframing & Technical Architecture 4 hrs ▼

Sketch your UI flows and establish the technical architecture before coding begins. Wireframes give AI a visual spec to work from. Basic architecture decisions (data storage, app structure) made now prevent costly rework later.

Key Topics
Lo-fi wireframes Navigation flow Data persistence options MVVM pattern basics App architecture diagram
Resources
🔧
Whimsical or Balsamiq (free tier) for wireframes
whimsical.com · No design skill needed · Draw boxes and arrows
📄
Apple Human Interface Guidelines — MacOS section
Free · developer.apple.com/design/human-interface-guidelines · Layout conventions
🎯 Deliverable: Wireframe for each screen + one-page architecture diagram. Attach both to your PRD.
⚡ AI Prompt Strategy

"Based on my PRD [paste], recommend a simple app architecture for a solo developer using SwiftUI. Explain the data storage options available (UserDefaults, CoreData, SQLite, JSON files) and which you'd recommend for my use case and why."

AI-Assisted Build

Weeks 7–12 · ~5 hrs/week
Core Build Phase

This is where you build the app — primarily by prompting AI. Your job is director, not typist. You'll write precise prompts using your PRD, review generated code, understand it well enough to debug, and iterate. Commit to Git after every working feature.

✓
Week 07 Project Scaffold & App Shell 5 hrs ▼

Set up the Xcode project structure, configure the app target for MacOS only, and build the skeleton navigation (windows, menus, tab views). Get a blank-but-working app running with proper file organisation.

Key Topics
Xcode project structure App entry point NavigationSplitView MacOS window config Menu bar setup
Resources
🎓
SwiftUI Masterclass — macOS sections
Udemy · Robert Petras · Search for macOS-specific content · ~2 hrs
🔧
Claude / ChatGPT for code generation
Use your PRD as context in every prompt session
🎯 Deliverable: MacOS app that launches, shows your main navigation structure, and is committed to GitHub
⚡ AI Prompt Strategy

"I'm building a MacOS SwiftUI app. Here is my PRD: [paste]. Generate the Xcode project structure and the main App file with a NavigationSplitView. The sidebar should have [X sections]. Explain what each piece of code does as inline comments."

✓
Week 08–09 Core Feature 1 — Full Implementation 10 hrs ▼

Pick the most important feature from your PRD and build it end-to-end. Include the UI, the data layer, and any business logic. Learn the pattern of: prompt → review → test → debug → commit. Repeat it for every subsequent feature.

Key Topics
Feature-by-feature prompting Data model implementation @Observable / ObservableObject Reading & debugging AI code Testing in Simulator
Resources
🎓
Build AI-Powered Apps with Claude & SwiftUI — search Udemy for current offerings
Udemy · Search "SwiftUI AI development" for updated courses
📄
Hacking with Swift — Paul Hudson (free reference)
hackingwithswift.com · Use as a lookup when AI output needs clarification
🎯 Deliverable: Feature 1 working end-to-end, all acceptance criteria from PRD passing
⚡ AI Prompt Strategy

"Here is the user story I want to implement: [paste from PRD]. My current app code is: [paste]. Generate the SwiftUI view and ViewModel to fulfil this story. Add comments. Then tell me how to test it manually." When something breaks: "Here is the error I'm seeing: [paste]. Here is the relevant code: [paste]. Diagnose the issue and provide a fix."

✓
Week 10–11 Core Features 2 & 3 — Iterative Build 10 hrs ▼

Apply the same prompt-review-test-commit loop to your remaining core features. By now you'll be faster and more confident reading code. Focus on integration — making features work together, not just in isolation.

Key Topics
Feature integration Data sharing between views Error handling Persistence testing UI polish pass
🎯 Deliverable: All Must-Have features from your PRD working and committed
⚡ AI Prompt Strategy

"I've now built Feature 1 and Feature 2 separately. I need them to share data through a central data store. Here is my current code: [paste]. Refactor this to use a shared EnvironmentObject or @Observable class so both features can read and write the same data."

✓
Week 12 Data Persistence & Settings 4 hrs ▼

Ensure data survives app restarts. Implement user preferences and settings. This is often overlooked until late but is critical for a usable app. Choose the right persistence layer (UserDefaults for settings, Core Data or JSON for structured data).

Key Topics
UserDefaults JSON encoding/decoding Core Data basics Preferences window App sandboxing
🎯 Deliverable: Data persists across app launches; settings panel working
⚡ AI Prompt Strategy

"My app stores [data type]. I need it to persist when the app closes. Based on my data model [paste], recommend whether to use UserDefaults, JSON file storage, or Core Data. Then implement the recommended approach."

Polish & Testing

Weeks 13–14 · ~4 hrs/week
Quality Gate

The app works — now make it feel good. MacOS users have high expectations for native app behaviour. Polish the UX, handle errors gracefully, and test thoroughly. Use AI to generate test cases from your acceptance criteria.

✓
Week 13 UX Polish & MacOS Native Behaviours 4 hrs ▼

Align your app with MacOS conventions — keyboard shortcuts, right-click menus, window management, dark mode support. Small native behaviours make the difference between an app that feels built and one that feels hacked together.

Key Topics
Keyboard shortcuts Context menus Dark/Light mode Toolbar customisation Accessibility basics Loading states
Resources
📄
Apple Human Interface Guidelines — Inputs, Menus, Windows
Free · developer.apple.com/design/human-interface-guidelines/
🎯 Deliverable: App supports dark mode, has at least 3 keyboard shortcuts, all loading states handled
⚡ AI Prompt Strategy

"Review my app code [paste] and suggest 5 MacOS-specific UX improvements I'm missing — things a native Mac user would expect. Then implement the keyboard shortcut for [action]."

✓
Week 14 Testing, Bug Fixing & Error Handling 5 hrs ▼

Systematically test every acceptance criterion from your PRD. Use AI to generate test cases you wouldn't think of. Fix bugs. Add proper error messages so the app never silently fails. Test edge cases: empty states, large data, unexpected inputs.

Key Topics
Manual test scripts Error handling patterns Empty state UI Edge case testing Bug tracking (simple log)
🎯 Deliverable: Test log showing all PRD acceptance criteria tested and passing; bug fix log documented
⚡ AI Prompt Strategy

"Here are my acceptance criteria: [paste from PRD]. Generate a manual test script with specific steps, inputs, and expected outcomes for a tester with no technical knowledge. Include at least 3 edge cases per feature."

Release

Weeks 15–16 · ~3 hrs/week
Finish Line

Since you're not submitting to the App Store, distribution is straightforward. Package the app, document it, and make it easy to install on any Mac you own. Create release documentation and a simple changelog.

✓
Week 15 Build, Sign & Package for Distribution 3 hrs ▼

Build a release version of your app, add an app icon, and package it as a .app bundle or .dmg for easy installation. Since you're not going to the App Store, you can use a free Apple Developer account and ad-hoc distribution. Sign the app to avoid Gatekeeper warnings on your own machines.

Key Topics
Release build config App icon creation Code signing Archive & export DMG creation Notarisation (optional)
Resources
📄
Apple: Distributing Your App Outside the App Store
Free · developer.apple.com · Direct distribution guide
🔧
App Icon Generator — appicon.co
Free · Generate all required icon sizes from one image
🔧
create-dmg (open source CLI tool)
github.com/create-dmg/create-dmg · Package your app into a .dmg installer
🎯 Deliverable: A signed .app or .dmg that installs and runs cleanly on your Mac
⚡ AI Prompt Strategy

"Walk me through step by step how to archive and export a MacOS SwiftUI app from Xcode for direct distribution (not App Store). I want to share it as a .dmg. I have a free Apple Developer account."

✓
Week 16 Documentation, Changelog & Retrospective 3 hrs ▼

Write user documentation (even for personal use, this is good practice). Create a changelog. Do a proper project retrospective — what worked, what you'd do differently, what AI did well vs. poorly. Plan v2 if you want to continue.

Key Topics
README writing Semantic versioning User guide Project retrospective v2 feature backlog
🎯 Deliverable: README, user guide, CHANGELOG.md, and retrospective document. Ship v1.0. 🎉
⚡ AI Prompt Strategy

"I've built [describe app]. Generate a user guide for a non-technical user. Also generate a retrospective framework asking the right questions about what I learned, what AI did well in the build process, and what to improve in v2."
